Goal

What Stage 1 sets out to prove

Build a working ImGui + OpenGL3 + GLFW window that detects the OAK-D-Lite device and displays its status in a sidebar — every line authored from scratch, no copied demo code.

Stage 1 does not open a pipeline. It only enumerates the device over USB and reports its state. This forces you to understand what the device looks like before a pipeline is running — which is the normal starting state.

What We Built

The running application

Window

  • GLFW window, 1280×720
  • OpenGL 3.3 core profile context
  • ImGui docking layout enabled
  • Fixed 60fps render loop

Sidebar (220px, left)

  • Green bullet + "Connected" when OAK is plugged in
  • Red bullet + "No device" when unplugged
  • MX ID of the connected device
  • State: "Unbooted (ready)" or "Booted / in use"

The OakDevice wrapper calls poll() once per frame. It never throws — any depthai exception is caught internally and the device is marked disconnected.

Key Concepts

What Stage 1 teaches you

X_LINK_UNBOOTED — the normal starting state

When the OAK is plugged in but no pipeline has been uploaded, it is in the Unbooted state. This is correct and expected. The device is visible to the host over USB, but the VPU is idle. Nothing is wrong.

Once you call dai::Device device(pipeline), the VPU receives the graph and enters the Booted state. Only one process can hold a Booted device at a time.

Two separate enumeration calls

depthai has two different queries and they live on different classes:

  • All connected devices (plugged in, any state): dai::XLinkConnection::getAllConnectedDevices()
  • Available devices (unbooted, not claimed): dai::DeviceBase::getAllAvailableDevices()

To check if a specific device is available, compare by MX ID — don't just check if the available list is non-empty.

ImGui immediate mode

Every frame, you call ImGui::Text(), ImGui::Bullet(), etc. to describe what should appear. ImGui draws it and discards the description. There is no retained widget tree. If you stop calling a function, that widget disappears.

The render loop shape

The loop runs at ~60fps: poll GLFW events → clear OpenGL → start ImGui frame → call your UI functions → render ImGui → swap buffers. The device poll happens once per frame, before UI.

Annotated Code

The most important patterns from Stage 1

Device.cpp — enumeration + no-throw contract

void OakDevice::poll() {
    try {
        // getAllConnectedDevices is on XLinkConnection, NOT DeviceBase
        const auto connected = dai::XLinkConnection::getAllConnectedDevices();
        if (connected.empty()) {
            connected_ = false;
            info_ = {};
            return;
        }

        const auto& first = connected[0];
        connected_ = true;
        info_.mxId = first.getMxId();

        // Separately check if this specific device is available (not claimed)
        const auto available = dai::DeviceBase::getAllAvailableDevices();
        info_.available = std::any_of(available.begin(), available.end(),
            [&](const dai::DeviceInfo& d) { return d.getMxId() == info_.mxId; }
        );
    } catch (const std::exception& e) {
        connected_ = false;
        info_ = {};
        fprintf(stderr, "Device poll error: %s\n", e.what());
    } catch (...) {
        connected_ = false;
        info_ = {};
    }
}

Sidebar.cpp — status indicator with color

if (device.isConnected()) {
    // PushStyleColor changes the color of everything until PopStyleColor
    ImGui::PushStyleColor(ImGuiCol_Text, ImVec4(0.2f, 0.9f, 0.3f, 1.0f)); // green
    ImGui::Bullet();       // use Bullet(), not "●" — default font is ASCII-only
    ImGui::SameLine();
    ImGui::Text("Connected");
    ImGui::PopStyleColor();

    ImGui::TextDisabled("MX ID");
    ImGui::TextWrapped("%s", info.mxId.c_str());

    // Human-readable state, not raw DeviceInfo.toString()
    ImGui::TextColored(..., info.available ? "Unbooted (ready)" : "Booted / in use");
}

Gotchas & Lessons Learned

Everything that bit us during Stage 1

ImGui docking is not in the master/release branch

ImGuiConfigFlags_DockingEnable and DockSpaceOverViewport only exist in the docking branch. Standard release zips from the GitHub releases page do not include them. You'll get compile errors about undeclared identifiers if you use the wrong version.

Fix: Download from github.com/ocornut/imgui/archive/refs/heads/docking.zip

depthai v2.17.3 is missing libusb-1.0.dll on Windows

The v2.17.3 prebuilt doesn't bundle libusb-1.0.dll. Without it, USB device enumeration silently fails — the device shows up in Device Manager but is invisible to the app. No error, just "No device" forever.

Fix: Use depthai-core v2.32.0 or later. It bundles libusb-1.0.dll and the POST_BUILD step copies it next to the executable automatically.

getAllConnectedDevices is NOT on dai::DeviceBase

The API is split across two classes. getAllConnectedDevices() is on dai::XLinkConnection and requires #include <depthai/xlink/XLinkConnection.hpp>. getAllAvailableDevices() is on dai::DeviceBase.

ImGui's default font (ProggyClean) is ASCII-only

Unicode characters like ● (U+25CF) render as ?. Use ImGui::Bullet() instead. To use Unicode you must load a custom font that covers the required code points.

Open VS2022 with File → Open → Folder, not Open Solution

CMake projects must be opened as a folder. VS2022 detects CMakePresets.json automatically. Opening a .sln file from a different project will give confusing build errors.

Old DLLs can survive a rebuild

When switching from depthai v2.17.3 to v2.32.0, the old DLLs may remain in the output directory until a clean rebuild triggers the POST_BUILD copy step. If the app behaves unexpectedly after changing deps, do a full clean rebuild.

The available check must be per-device, not global

Checking !available.empty() answers "is any device available?" — not "is this specific device available?". Use std::any_of comparing by MX ID to check the device you actually found.
