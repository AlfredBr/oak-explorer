Stage Overview

From sensor to pixel on screen — 5 steps

1

Build a Pipeline on the host

Create a dai::Pipeline. Add a ColorCamera node and an XLinkOut node. Wire them together. Nothing runs yet — this is just a description.

2

Boot the device

dai::Device device(pipeline) uploads the graph to the VPU over USB and starts execution. Sidebar shows "Booted / in use".

3

Receive frames on the host

Call device.getOutputQueue("rgb"). Each call to queue->tryGet() returns the latest ImgFrame — or nullptr if none is ready. Non-blocking.

4

Upload to an OpenGL texture

Extract raw pixel bytes from the ImgFrame and call glTexSubImage2D() to push them to the GPU. One texture object, updated every frame.

5

Render with ImGui::Image()

Pass the OpenGL texture ID to ImGui::Image(). ImGui draws it scaled to the window. Live video in your harness.

Concept 1 — Booting the Pipeline

What actually happens when you call dai::Device device(pipeline)

HOST

You describe the graph

Create a dai::Pipeline. Add nodes. Wire them. Nothing is running — this is a specification, like a recipe, not a meal.

USB

Pipeline is serialised and uploaded

dai::Device device(pipeline) serialises the graph and sends it to the OAK over USB. Device transitions Unbooted → Booted.

VPU

The OAK executes the graph autonomously

The Intel Myriad X VPU runs your nodes independently of the host CPU. The color camera starts capturing. Frames flow through the graph. Your PC is not involved.

Q

Frames accumulate in an output queue

The XLinkOut node ships frames over USB into a named queue on the host. Your render loop calls tryGet() each frame — non-blocking, returns nullptr if empty.

Key insight

The OAK is not a webcam. It's a small computer. You program it. The pipeline is a program that runs on the device, not on your PC. Your PC just reads the output.

ColorCamera — Source node

Drives the RGB sensor. Configures resolution, frame rate, and pixel format. Produces frames continuously once the pipeline is booted.

  • setPreviewSize(1280, 720) — output resolution for the preview stream
  • setInterleaved(true) — interleaved RGBRGBRGB… layout — required for GL_RGB
  • setColorOrder(RGB) — R, G, B byte order to match OpenGL's GL_RGB format
  • setFps(30) — target frame rate
Critical — setInterleaved must be true

setInterleaved(false) outputs planar CHW layout: all R bytes, then all G bytes, then all B bytes. OpenGL's GL_RGB expects interleaved RGBRGBRGB… bytes. Using false renders as a 3×3 tiled grayscale grid — a confusing symptom. Always use true when uploading to an OpenGL texture.

XLinkOut — Output node

A pipeline tap — ships data from the OAK to the host over USB. Every stream needs one. You give it a name; the host uses that same name to open the receive queue.

  • setStreamName("rgb") — the name used on the host side
  • cam->preview.link(xout->input) — wires the camera output into XLinkOut

Building the pipeline

dai::Pipeline pipeline;

auto cam = pipeline.create<dai::node::ColorCamera>();
cam->setPreviewSize(1280, 720);
cam->setInterleaved(true);   // interleaved RGBRGB… — required for GL_RGB
cam->setColorOrder(dai::ColorCameraProperties::ColorOrder::RGB);
cam->setFps(30);

auto xout = pipeline.create<dai::node::XLinkOut>();
xout->setStreamName("rgb");

cam->preview.link(xout->input); // wire camera → XLinkOut

Concept 3 — Output Queues & ImgFrame

How the host receives frames from the OAK

The OAK produces frames at 30fps independently of your render loop. They accumulate in a fixed-size buffer on the host. Your loop calls tryGet() once per frame — non-blocking, returns nullptr if the queue is empty.

Two calls: setup once, poll every frame

// Once at startup — get the queue by the stream name we gave XLinkOut
auto queue = device.getOutputQueue("rgb", 4, false);
//                                   name  maxSize  blocking

// Every render frame — non-blocking poll
auto frame = queue->tryGet<dai::ImgFrame>();
if (frame) {
    // new frame — upload to texture
}

maxSize=4 means the queue holds up to 4 frames before the oldest is dropped. blocking=false means tryGet() never stalls your render loop.

What's inside an ImgFrame

  • getData() — raw pixel bytes. For BGR, 3 bytes per pixel, width×height×3 total. This is what you hand to OpenGL.
  • getWidth() / getHeight() — frame dimensions. Should match setPreviewSize().
  • getTimestamp() — capture time on the device clock. Useful for sync in Stage 3+.
  • getSequenceNum() — frame counter. If this jumps, frames were dropped in the queue.

Concept 4 — OpenGL Texture Upload

Getting pixel bytes from CPU RAM onto the GPU

Every frame, getData() gives you a pointer to raw BGR bytes in CPU RAM. You hand those to OpenGL with glTexSubImage2D() and it copies them to GPU VRAM. One texture object lives for the lifetime of the app — you overwrite it each frame.

Phase 1 — Create the texture once at startup

GLuint texId;
glPixelStorei(GL_UNPACK_ALIGNMENT, 1);  // RGB rows may not be 4-byte aligned
glGenTextures(1, &texId);
glBindTexture(GL_TEXTURE_2D, texId);
glTexParameteri(GL_TEXTURE_2D, GL_TEXTURE_MIN_FILTER, GL_LINEAR);
glTexParameteri(GL_TEXTURE_2D, GL_TEXTURE_MAG_FILTER, GL_LINEAR);

// Allocate GPU memory (nullptr = empty for now)
glTexImage2D(GL_TEXTURE_2D, 0, GL_RGB,
    width, height, 0, GL_RGB, GL_UNSIGNED_BYTE, nullptr);
glBindTexture(GL_TEXTURE_2D, 0);
glPixelStorei(GL_UNPACK_ALIGNMENT, 4);  // restore default

Phase 2 — Upload each new frame

auto frame = queue->tryGet<dai::ImgFrame>();
if (frame) {
    glBindTexture(GL_TEXTURE_2D, texId);
    glTexSubImage2D(GL_TEXTURE_2D, 0,  // target, mip level
        0, 0,                          // x, y offset
        width, height,                 // dimensions
        GL_RGB, GL_UNSIGNED_BYTE,      // format, type
        frame->getData().data());      // pixel pointer
    glBindTexture(GL_TEXTURE_2D, 0);
}

OpenGL API — what each call actually does

glGenTextures(1, &texId)

Asks OpenGL to reserve a texture slot and hand you back an integer ID (GLuint). The first argument is how many textures to create at once — we only need one. Nothing is allocated in VRAM yet; this is just bookkeeping.

glBindTexture(GL_TEXTURE_2D, texId)

OpenGL is a state machine — you don't pass a texture to every function, you bind it first and all subsequent texture calls target it. GL_TEXTURE_2D is the target slot (a 2D image, as opposed to a cube map or 3D texture). You bind before creating, before setting parameters, and before each upload.

glTexParameteri — MIN_FILTER and MAG_FILTER

Controls how OpenGL samples the texture when drawn at a size different from its native resolution. MIN_FILTER applies when shrunk (minified); MAG_FILTER applies when enlarged (magnified). GL_LINEAR blends neighbouring pixels for a smooth result. The alternative, GL_NEAREST, snaps to the closest pixel — useful for pixel art but blocky for camera video.

glTexImage2D(target, mip, internalFormat, w, h, border, format, type, data)

Allocates VRAM and optionally uploads initial pixel data. Key parameters:

  • mip = 0 — base mip level. We don't use mipmaps for live video.
  • internalFormat = GL_RGB — how OpenGL stores it in VRAM: 3 channels, 8 bits each.
  • format = GL_RGB — the byte order of the data. Must match setColorOrder(RGB) and setInterleaved(true).
  • data = nullptr — allocates memory without uploading anything yet. The first real frame fills it via glTexSubImage2D.

glTexSubImage2D(target, mip, x, y, w, h, format, type, data)

Updates a region of an already-allocated texture in place — no reallocation. This is what you call every frame. It's much faster than glTexImage2D because the VRAM is already reserved.

  • x=0, y=0 — top-left corner, we replace the full texture each frame.
  • GL_RGB — matches ColorOrder::RGB and setInterleaved(true). Note: GL_BGR is not valid in OpenGL 3.3 core profile.
  • getData().data() — returns a raw uint8_t* pointer to the frame's pixel buffer in CPU RAM.

The texture ID is the bridge to ImGui

GLuint texId is an integer OpenGL gave you. ImGui accepts an ImTextureID (a void*). Cast it: (ImTextureID)(uintptr_t)texId. The ImGui OpenGL backend uses it to bind and draw your texture. That's the entire bridge between OpenGL and ImGui.

Concept 5 — ImGui::Image()

Rendering your OpenGL texture inside an ImGui window

After all the setup, displaying the frame is two lines. ImGui::Image() takes the texture ID and a display size — it emits a textured rectangle into ImGui's draw list and the OpenGL backend does the rest.

// Inside an ImGui window, every frame:
ImGui::Image(
    (ImTextureID)(uintptr_t)texId,  // GL texture ID cast to void*
    ImVec2(width, height)           // display size in pixels
);

What ImGui::Image() actually does

  1. Emits a textured quad into ImGui's draw list — a rectangle UV-mapped 0→1 across the whole texture.
  2. At render time, the OpenGL backend casts ImTextureID back to GLuint, calls glBindTexture, and issues a draw call.
  3. The GPU samples your texture and fills the rectangle with your live camera frame.

Sizing — fill the window width

// Scales to available window width, preserves 16:9 aspect ratio
float avail = ImGui::GetContentRegionAvail().x;
float aspect = (float)height / (float)width;
ImGui::Image((ImTextureID)(uintptr_t)texId, ImVec2(avail, avail * aspect));

Why the double cast?

ImTextureID is void* — pointer-sized — so ImGui works with any backend (OpenGL, Vulkan, DirectX, Metal) without depending on any of them. GLuint is 32-bit. The cast first widens to uintptr_t (pointer size), then to void*. The backend reverses it. This is the standard pattern in every ImGui + OpenGL example.

Full render loop — all 5 concepts together

// 1. Poll for a new camera frame and upload to GPU
auto frame = queue->tryGet<dai::ImgFrame>();
if (frame) {
    glBindTexture(GL_TEXTURE_2D, texId);
    glTexSubImage2D(GL_TEXTURE_2D, 0, 0, 0,
        width, height, GL_RGB, GL_UNSIGNED_BYTE,
        frame->getData().data());
}

// 2. Render the ImGui frame
ImGui::NewFrame();
ImGui::Begin("Camera");
float avail = ImGui::GetContentRegionAvail().x;
float aspect = (float)height / (float)width;
ImGui::Image((ImTextureID)(uintptr_t)texId, ImVec2(avail, avail * aspect));
ImGui::End();
ImGui::Render();
// ... swap buffers

Stage 2 Architecture

What's new, what changes, and what stays the same

oak-explorer/
├── src/
│   ├── main.cpp             ← modified — owns dai::Device and texture lifetime
│   ├── oak/
│   │   ├── Device.h/cpp      ← modified — adds pipeline build + device open/close
│   │   └── CameraStream.h/cpp ← new    — owns texture, polls queue, uploads to GL
│   └── ui/
│       ├── Sidebar.cpp       ← modified — shows "Booted / in use" when streaming
│       └── StreamView.h/cpp   ← new    — ImGui window with ImGui::Image()

CameraStream — new oak/ class

Owns everything depthai-related for Stage 2: builds the pipeline, opens dai::Device, gets the output queue, creates and owns the GL texture, and exposes a simple polling interface.

  • poll() — calls tryGet(), uploads frame if available
  • textureId() — returns the GLuint for ImGui
  • width() / height() — frame dimensions
  • isStreaming() — true once the device is open

StreamView — new ui/ class

Pure UI — knows nothing about depthai. Opens an ImGui window, calls ImGui::Image() with the texture ID from CameraStream, scales to fill available width. Takes const CameraStream& — read-only.

What stays the same

  • OakDevice — still polls for enumeration, still drives the Sidebar. The pipeline device open is separate.
  • Sidebar — unchanged shape; will now show "Booted / in use" once the pipeline starts.
  • main.cpp render loop — same structure, adds stream.poll() and ui::renderStreamView(stream).

Design principle — depthai and ImGui never touch

CameraStream owns depthai. StreamView owns ImGui. They communicate through a read-only interface: textureId(), width(), height(), isStreaming(). This boundary holds through all six stages.

Annotated Code

CameraStream.cpp — line by line

open() — build pipeline and create GL texture

void CameraStream::open() {
  try {
    // --- Build the pipeline (host-side description only) ---
    dai::Pipeline pipeline;

    auto cam = pipeline.create<dai::node::ColorCamera>();
    cam->setPreviewSize(width_, height_);
    // true = interleaved RGBRGB… — GL_RGB expects this byte order
    // false = planar CHW (all R, then G, then B) — causes 3×3 tiled display
    cam->setInterleaved(true);
    // GL_BGR is NOT valid in OpenGL 3.3 core profile — must use RGB
    cam->setColorOrder(dai::ColorCameraProperties::ColorOrder::RGB);
    cam->setFps(30);

    auto xout = pipeline.create<dai::node::XLinkOut>();
    xout->setStreamName("rgb");
    cam->preview.link(xout->input);  // camera.preview → XLinkOut

    // --- Boot the device (uploads graph to VPU over USB) ---
    device_ = std::make_unique<dai::Device>(pipeline);

    // --- Get the output queue (non-blocking, max 4 frames buffered) ---
    queue_ = device_->getOutputQueue("rgb", 4, false);

    // --- Create GL texture ---
    // Set alignment=1: RGB rows at 1280px are 3840 bytes — not always 4-byte aligned
    glPixelStorei(GL_UNPACK_ALIGNMENT, 1);
    glGenTextures(1, &texId_);
    glBindTexture(GL_TEXTURE_2D, texId_);
    glTexParameteri(GL_TEXTURE_2D, GL_TEXTURE_MIN_FILTER, GL_LINEAR);
    glTexParameteri(GL_TEXTURE_2D, GL_TEXTURE_MAG_FILTER, GL_LINEAR);
    // Allocate VRAM without uploading pixels yet (nullptr)
    glTexImage2D(GL_TEXTURE_2D, 0, GL_RGB,
        width_, height_, 0, GL_RGB, GL_UNSIGNED_BYTE, nullptr);
    glBindTexture(GL_TEXTURE_2D, 0);
    glPixelStorei(GL_UNPACK_ALIGNMENT, 4);  // restore OpenGL default

    streaming_ = true;
  } catch (const std::exception& e) {
    fprintf(stderr, "CameraStream::open failed: %s\n", e.what());
    close();  // clean up any partial state
  }
}

close() — destruction order matters

void CameraStream::close() {
    streaming_ = false;
    queue_.reset();   // release queue BEFORE device — queue holds a reference
    device_.reset();  // destructor stops the pipeline on the VPU
    if (texId_ != 0) {
        glDeleteTextures(1, &texId_);  // must be called while GL context is active
        texId_ = 0;
    }
}
Key: call close() before GLFW teardown

glDeleteTextures requires an active OpenGL context. In main.cpp, call stream.close() before ImGui::DestroyContext() and glfwDestroyWindow().

poll() — called every render frame

void CameraStream::poll() {
    if (!streaming_) return;
    try {
        // tryGet() is non-blocking — returns nullptr if no new frame yet
        auto frame = queue_->tryGet<dai::ImgFrame>();
        if (frame) {
            // Update texture in-place — no VRAM reallocation (glTexSubImage2D)
            glBindTexture(GL_TEXTURE_2D, texId_);
            glTexSubImage2D(GL_TEXTURE_2D, 0, 0, 0,
                width_, height_,
                GL_RGB, GL_UNSIGNED_BYTE,
                frame->getData().data());
            glBindTexture(GL_TEXTURE_2D, 0);
        }
    } catch (const std::exception& e) {
        fprintf(stderr, "CameraStream::poll error: %s\n", e.what());
        streaming_ = false;  // stop trying on error
    }
}

Gotchas & Lessons Learned

Every bug we hit implementing Stage 2 — and the fix

1 — WINGDIAPI / APIENTRY redefinition (~100 C2086 errors)

Symptom: After adding #include <GL/GL.h> to CameraStream.cpp, the build explodes with ~100 errors in GL/GL.h: "C2086: 'int WINGDIAPI': redefinition".

Cause: Windows SDK's GL/GL.h defines WINGDIAPI and APIENTRY. But #include <depthai/depthai.hpp> (which appears earlier) pulls in Windows headers that also define them. Second definition = error.

Fix: Replace #include <GL/GL.h> with #include <GLFW/glfw3.h>. GLFW guards these macros correctly and covers all modern OpenGL constants (the Windows SDK GL.h only covers OpenGL 1.1).

2 — 3×3 tiled grayscale image

Symptom: App builds and runs, device connects, but the Camera window shows a 3×3 grid of small grayscale versions of the scene.

Cause: setInterleaved(false) outputs planar CHW data: all R bytes, then all G bytes, then all B bytes. OpenGL's GL_RGB expects interleaved RGBRGBRGB… data. OpenGL misinterprets the planar buffer as 3 side-by-side grayscale panels.

Fix: Change setInterleaved(false) → setInterleaved(true).

3 — 3–5 fps frame rate

Symptom: App runs correctly but frame rate is 3–5 fps instead of 30.

Cause: OakDevice::poll() called dai::XLinkConnection::getAllConnectedDevices() and dai::DeviceBase::getAllAvailableDevices() every render frame. USB enumeration costs ~100–200ms per call. Both calls per frame = 200–400ms overhead = 3–5 fps cap.

Fix: Throttle OakDevice::poll() to 1 Hz using std::chrono::steady_clock. Device connect/disconnect is a human-speed event — polling at 1 Hz is more than fast enough.

4 — Camera window docked inside sidebar

Symptom: The new "Camera" window appears inside the sidebar panel, not as a separate floating window.

Cause: ImGui saves docking layout to imgui.ini. On first launch (or after any previously saved layout), new windows may auto-dock. Additionally, imgui.ini from Stage 1 didn't include the Camera window, so ImGui placed it in the nearest available dock area.

Fix: (1) Delete imgui.ini to clear the saved layout. (2) Add ImGui::SetNextWindowPos(ImVec2(240, 10), ImGuiCond_FirstUseEver) and SetNextWindowSize(ImVec2(900, 540), ImGuiCond_FirstUseEver) at the top of renderStreamView() to set a sensible default position for first use.

5 — GL_UNPACK_ALIGNMENT not restored

Symptom: Code review catch — not a visible bug at 1280px width, but correctness issue.

Cause: After glPixelStorei(GL_UNPACK_ALIGNMENT, 1), the alignment was never restored to the default value of 4. Any subsequent OpenGL texture operations in the app (e.g., ImGui's own fonts) would inherit the alignment-1 setting.

Fix: Add glPixelStorei(GL_UNPACK_ALIGNMENT, 4) immediately after the glTexImage2D call.

Pattern — GL_BGR is not valid in OpenGL 3.3 core profile

GL_BGR is an OpenGL extension (part of OpenGL 1.2+ compatibility profile). It is not available in the core profile used by this project. Always use GL_RGB and match it with ColorOrder::RGB on the camera side. The performance difference is zero — the reorder happens in silicon on the GPU anyway.
